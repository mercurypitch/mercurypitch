# URL Hash Routing Plan

## Context

The app currently has **no router** — navigation is purely signal-based via `activeTab` in
`ui-store.ts`. Share URLs generated by the app (`/uvr/session/:id`, `/share?...`) return 404
because there is no client-side routing and no SPA fallback in Apache.

Hash-based routing fixes this without any server config changes, since everything after `#`
stays client-side. The server always serves `index.html` for `/`, and the hash is read by JS.

## Current State

| Aspect | Status |
|--------|--------|
| Router | None |
| Navigation | `activeTab` signal (`'practice' | 'editor' | 'settings' | 'vocal-analysis' | 'community' | 'leaderboard' | 'vocal-challenges' | 'uvr'`) |
| View switching | 8 `<Show when={activeTab() === 'X'}>` blocks in App.tsx |
| Share URLs | `/uvr/session/:id` and `/share?type=...&id=...` — both 404 on direct access |
| `window.location` usage | Only `share-url.ts` uses `window.history.replaceState` for query-param melody sharing |

## Hash Format

```
#/tab-name                     → top-level tab navigation
#/uvr/session/:sessionId      → direct link to UVR session
#/share?type=T&id=I           → community share link
```

Tab name mapping matches the `ActiveTab` type values: `practice`, `editor`, `settings`,
`vocal-analysis`, `community`, `leaderboard`, `vocal-challenges`, `uvr`.

## Architecture

A single `createHashRouter` primitive (new file: `src/lib/hash-router.ts`) that:

1. **Reads** `window.location.hash` on page load → derives initial `activeTab` and sub-state
2. **Watches** `hashchange` event → updates signals when user presses back/forward
3. **Provides** a `navigateTo(hash)` function that sets `window.location.hash` (triggering
   `hashchange` instead of directly mutating signals)

### Bidirectional sync

```
                 hashchange event
window.location.hash ──────────────→ hashRouter signals ──→ setActiveTab / sub-views
       ↑                                                         │
       │                                                         │
       └─────────────────────────────────────────────────────────┘
                    navigateTo(hash) ←─ app calls on user action
```

This keeps a single source of truth (the URL hash) and avoids two-way binding races.

### Key insight: don't fight existing signals

`activeTab` and `UvrPanel.currentView` stay as they are. The hash router only:
- Sets them on initial load (if a hash is present)
- Updates the hash when they change (via an `effect`)
- Listens for external hash changes (back/forward buttons, direct navigation)

## Implementation Steps

### Phase 1: Core hash router (`src/lib/hash-router.ts`)

Create a lightweight hash routing module:

```ts
// Types
export type HashRoute =
  | { type: 'tab'; tab: ActiveTab }
  | { type: 'uvr-session'; sessionId: string }
  | { type: 'share'; shareType: string; shareId: string }
  | { type: 'unknown' }

// Parse current hash → route
export function parseHash(hash: string): HashRoute

// Build hash string from route
export function buildHash(route: HashRoute): string

// Navigate to a hash (updates window.location.hash)
export function navigateTo(route: HashRoute): void
```

Parsing rules:
- `#/practice` → `{ type: 'tab', tab: 'practice' }`
- `#/uvr/session/abc123` → `{ type: 'uvr-session', sessionId: 'abc123' }`
- `#/share?type=melody&id=xyz` → `{ type: 'share', shareType: 'melody', shareId: 'xyz' }`
- `#/uvr` → `{ type: 'tab', tab: 'uvr' }`
- Anything unrecognized → `{ type: 'unknown' }`

### Phase 2: Wire into App.tsx

In `AppShell`:

1. **On mount**: Parse `window.location.hash`. If it maps to a known route, set the
   corresponding `activeTab` and any sub-state.

2. **Hash effect**: `createEffect` that watches `activeTab()` and updates
   `window.location.hash` to `#/tab-name` (using `history.replaceState` to avoid creating
   history entries on every tab switch — only `navigateTo` creates entries).

3. **hashchange listener**: On external hash change (back/forward), parse the new hash and
   update `activeTab` accordingly.

```ts
// In AppShell
onMount(() => {
  const route = parseHash(window.location.hash)
  if (route.type === 'tab') setActiveTab(route.tab)
  // Phase 3 handles uvr-session and share routes

  window.addEventListener('hashchange', () => {
    const newRoute = parseHash(window.location.hash)
    if (newRoute.type === 'tab') setActiveTab(newRoute.tab)
  })
})

createEffect(() => {
  const hash = `#/${activeTab()}`
  if (window.location.hash !== hash) {
    history.replaceState(null, '', hash)
  }
})
```

### Phase 3: Fix Share URLs

**UvrResultViewer.tsx**: Change share URL from:
```
/uvr/session/uvr-session-1234567890123
```
to:
```
#/uvr/session/uvr-session-1234567890123
```

**CommunityShare.tsx**: Change share URL from:
```
/share?type=melody&id=abc
```
to:
```
#/share?type=melody&id=abc
```

**SharePage.tsx**: Update to read hash params instead of `window.location.search`.

### Phase 4: Handle UVR session deep link

When the hash router sees `#/uvr/session/:id`:
1. Set `activeTab` to `'uvr'`
2. Pass `initialSessionId` to `UvrPanel` so it can load that session directly

### Phase 5: Add linkable sessions to UvrPanel

Add a "Copy Link" button to each session in the history view, using `#/uvr/session/:id`.

## What This Does NOT Change

- **No router dependency added** — stays zero-dependency
- **`activeTab` signal stays** — unchanged, just gets set by hash and vice versa
- **Tab bar buttons** — still call `handleTabChange()` as before; the hash updates as a side effect
- **UvrPanel internal `currentView`** — unchanged; the router only sets it on initial load
- **Apache config** — no changes needed (hash never hits the server)
